AI Agents Revolutionize LLM Research on Single-GPU Setups Overnight

Karpathy's autoresearch deploys autonomous swarms to edit, train, and evolve nanochat models without human intervention.

karpathy/autoresearch Python 18.5k stars

Imagine waking up to a log of dozens of experiments, each tweaking a language model architecture, optimizer, or training recipe—and finding your nanochat LLM noticeably sharper. That's the promise of autoresearch, Andrej Karpathy's latest brainchild, which hands the reins of machine learning experimentation to AI agents running on a humble single GPU.

At its core, autoresearch automates the drudgery of research iteration. You start with a stripped-down training pipeline for nanochat, a bite-sized LLM akin to Karpathy's earlier nanoGPT. The setup lives in just three key files: the immutable prepare.py for data prep and utilities; the mutable train.py, which houses the GPT model, Muon+AdamW optimizer, and training loop; and program.md, a Markdown blueprint that "programs" your AI research organization.

Here's the magic: instead of manually hacking Python, you define high-level instructions in program.md. An AI agent—powered by frontier models—reads this context, proposes edits to train.py, trains for a brisk five minutes, evaluates perplexity or other metrics, and commits improvements if they stick. Rinse and repeat overnight. Discards pile up as failures, but keepers compound, potentially yielding a superior model by morning. No clusters required; a consumer GPU suffices.

This isn't mere hyperparameter tuning. Agents rewrite model code wholesale—altering layer counts, activations, or even inventing schedules. Karpathy's tongue-in-cheek README frames it as the dawn of "autonomous swarms of AI agents," mocking the "meat computers" of old with their pesky sleep cycles. Yet the technical insight is profound: by constraining the playground to nano-scale training, it sidesteps the compute barriers of big labs, democratizing automated discovery.

What problem does it solve? ML research is bottlenecked by human tedium—endless edit-train-eval cycles that take hours and intuition. Autoresearch offloads this to agents, accelerating solo exploration or prototyping. It's for tinkerers with a GPU who want to probe "what if" questions at machine speed: Does a custom attention variant beat rotary embeddings? Can a novel optimizer edge out AdamW?

Technically, it's elegant minimalism. Fixed data (tokenized TinyStories or similar) ensures reproducibility; short epochs keep cycles snappy; agent prompts evolve via your program.md tweaks, like adding specialist agents for architecture search or data augmentation. Early adopters are iterating on the "research org code" itself—spawning sub-agents for debate or ensemble evals.

Gaining explosive traction in days, autoresearch signals a paradigm shift. It blurs lines between researcher and tool, hinting at self-improving AI labs. Risks? Agent hallucinations could loop fruitlessly, but baselines mitigate this. For now, it's a playground for the future: program your agents, hit run, and let evolution do the rest. As Karpathy tweets, this is "how it all began."

Symphony-ts Ports OpenAI's Isolated AI Agents to TypeScript Workflows

Unofficial Node.js implementation automates issue resolution with per-task workspaces, agent runtime, and operator dashboards

OasAIStudio/symphony-ts · TypeScript · 376 stars

Builders grappling with AI coding agents now have a TypeScript-native option: symphony-ts, an unofficial port of OpenAI's Symphony project. Launched just days ago, it transforms ticket trackers into autonomous implementation pipelines, enforcing strict isolation per issue to prevent cross-contamination in repositories.

The core workflow is straightforward yet powerful. Symphony-ts scans a repository's WORKFLOW.md file— a YAML-like config specifying trackers like Linear via LINEAR_API_KEY. For each pulled issue, it provisions a dedicated workspace, spins up a compatible coding agent (e.g., codex app-server), and executes within those boundaries. Operators gain a web dashboard on port 4321 for real-time visibility, retries, and interventions. A high-trust preview flag (--acknowledge-high-trust-preview) is mandatory, underscoring its design for secure, internal use.

Installation is CLI-driven: npm install -g symphony-ts, then run from the repo root with symphony --acknowledge-high-trust-preview. No auto-generation of WORKFLOW.md; repos must own their workflows, templated as:

---
tracker:
  kind: linear
  api_key: $LINEAR_API_KEY

Recent v0.1.7 release polishes the experience:

  • Live dashboard updates and UI alignment with upstream observability.
  • Restructured server modules for better modularity.
  • Apache 2.0 licensing for open reuse.
  • Clarified auth/network docs and fixes for worker reconciliation.

These changes, from contributors like @Octane0411 and new face @Aubrey-M-ops, address early pain points in onboarding and reliability. Developers must supply Node.js >=22 and agent runtimes supporting app-server mode.

Why does this matter? In TS/Node ecosystems, where AI agents promise velocity but risk repo sprawl, symphony-ts delivers bounded autonomy. It bridges trackers to code without bespoke infra, letting teams delegate implementation while retaining control. Early traction hints at demand, but its value lies in taming agent chaos for production pipelines—not metrics.

For TypeScript-heavy teams, it's a signal to prototype: export your keys, draft WORKFLOW.md, and test isolation on non-critical issues. As agent tooling matures, expect integrations beyond Linear.

Local SOCKS5 Proxy Speeds Telegram Desktop Loading

Python tool intercepts traffic and routes via WebSocket to Telegram data centers

Flowseal/tg-ws-proxy · Python · 743 stars

Flowseal/tg-ws-proxy is a Python-based local SOCKS5 proxy server designed to accelerate Telegram Desktop by partially bypassing network restrictions. It runs on 127.0.0.1:1080, intercepting connections to Telegram IP addresses and extracting data center (DC) IDs from MTProto obfuscation packets.

The proxy then establishes TLS WebSocket connections to domains like kws{N}.web.telegram.org. If WebSocket fails due to redirects, it falls back to direct TCP. This mirrors hosts-file tweaks for web.telegram.org, speeding message loads, file downloads, and partial media.

Setup is straightforward:

  • Windows: Download TgWsProxy.exe from releases; it deploys a tray app with GUI config, logs, and tg://socks proxy links for Telegram Desktop.
  • Source: pip install -r requirements.txt, then python proxy/tg_ws_proxy.py --port 1080 (verbose logging via -v).

Options include custom ports and DC IPs like 2:149.154.167.220. The v1.0.3 release adds experimental Windows 7 support. At six days old, it targets users facing throttled Telegram access, improving reliability without full VPNs.

In restricted networks, it delivers measurable gains in load times for high-volume chats and files, filling a niche for desktop clients.

Builders targeting Base, Coinbase's Ethereum Layer 2 on the Optimism OP Stack, now have a streamlined path to sovereignty. The base/node GitHub repository delivers Docker Compose builds for spinning up full nodes on Base mainnet or Sepolia testnet, addressing the complexity of syncing L2 chains that depend on Ethereum L1 data.

At its core, base/node solves the orchestration headache for OP Stack deployments. Users supply an L1 full node RPC, beacon endpoint, and optional archiver via .env files—.env.mainnet for production, .env.sepolia for testing. A single command launches everything:

docker compose up --build

For testnet: NETWORK_ENV=.env.sepolia docker compose up --build.

It supports three execution clients: reth (default, with Flashblocks via RETH_FB_WEBSOCKET_URL), geth, and nethermind. Switch with CLIENT=reth (or geth/nethermind) prefixed to the Docker command. This flexibility accommodates varying performance needs, from reth's speed on NVMe RAID to geth's stability.

Hardware demands are production-grade. Minimum: multicore CPU, 32GB RAM (64GB advised), NVMe SSD with 2x chain size plus snapshots and 20% buffer. Base-org's setups use AWS i7i.12xlarge instances with ext4 on RAID 0 NVMe drives for archive (reth) or full (geth) nodes—benchmarks for serious operators.

The v0.14.9 release, pushed March 2026, flags as recommended for Base Geth nodes. Key updates: op-geth to v1.101609.1 (minor diffs in Ethereum-optimism's fork) and op-node to v1.16.7. Full changelog tracks Optimism upstream, ensuring sync with L2 security patches.

For builders, this matters because Base's low-cost, dev-friendly L2 thrives on decentralized validation. Running your own node verifies transactions independently, supports custom indexers, and enables MEV strategies without RPC reliance. With 68,705 stars signaling broad adoption, it lowers barriers for dApp teams, validators, and infra providers chasing Ethereum scaling. No more wrestling raw OP Stack configs—base/node packages it for rapid deployment, future-proofed for chain growth.

Project Aura Builds Polished ESP32 Air-Quality Station for Makers

Open-source firmware delivers comprehensive indoor monitoring with LVGL UI, web dashboard, and seamless Home Assistant integration on ESP32-S3.

21cncstudio/project_aura C Latest: v1.1.0 281 stars

Indoor air quality remains a blind spot for many smart homes, where bare sensor boards demand endless tinkering. Project Aura changes that. This open-source ESP32-S3 station packs professional-grade telemetry—PM0.5 through PM10, CO, CO2, VOC, NOx, temperature, humidity, absolute humidity, pressure, and HCHO—into a touch-friendly device aimed at builders seeking reliability over prototypes.

At its core, Aura fuses Sensirion SEN66 and SFA30 sensors with a smooth LVGL UI featuring night mode, custom themes, and status indicators. No soldering: it uses Grove and QT connectors for plug-and-play assembly. A local web setup portal handles Wi-Fi onboarding via AP mode and mDNS (http://aura.local), while a full dashboard at /dashboard offers live sensors, charts, events, settings sync, and even DAC control for GP8403 valves (0-10V output with auto-thresholds).

Version 1.1.0, released last month, introduces OTA firmware updates directly from the dashboard—upload .bin files via Settings/System, with lifecycle handling (begin/write/end/abort) and slot checks to preserve user configs. During OTA, the device UI shifts to a dedicated update screen, avoiding frozen impressions. Robust Safe Boot rolls back to the last-good config on crashes.

Firmware, written in C with PlatformIO, supports MQTT auto-discovery for Home Assistant, including ready-to-import dashboards. Optional DAC mode automates ventilation based on air thresholds. Builders access pin configs, multi-language UI, and a repo layout for easy customization.

A crowdfunding campaign on MakerWorld promises 3D-printable enclosures, wiring guides, and BOM details, lowering barriers for production. With 281 stars in two months, it shows steady maker interest.

For developers, Aura stands out by prioritizing polish: integrated web experiences in AP/STA modes, theme presets, and backlight controls make it deployable, not debuggable.

  • Hardware: Waveshare displays, Adafruit-compatible sensors.
  • Networking: mDNS, MQTT, OTA.
  • Extensibility: Open for contribs via GitHub Discussions.

This isn't another sensor sketch—it's firmware for devices that last.
