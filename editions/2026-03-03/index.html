Reverse-Engineered APIs Unlock Training on Apple's Neural Engine

Proof-of-concept demonstrates backpropagation directly on ANE hardware, bypassing CoreML's inference-only restrictions

maderix/ANE Objective-C 3.3k stars

In a breakthrough for Apple silicon enthusiasts, the maderix/ANE project enables training neural networks directly on the Apple Neural Engine (ANE)—the dedicated neural processing unit in every modern iPhone, iPad, and Mac—using reverse-engineered private APIs. This isn't inference via the sanctioned CoreML pipeline; it's full backpropagation, leveraging raw ANE compute without Metal or GPU involvement.

At its core, ANE exposes powerful matrix multiplication and convolution accelerators optimized for machine learning workloads. Yet Apple has long locked it down to inference-only through CoreML, leaving training to GPUs or CPUs. Developer maderix cracked this open over a weekend by tapping undocumented private frameworks like _ANEClient and _ANECompiler. These APIs, hidden in Apple's ecosystem, allow compiling custom ops, allocating ANE memory, and executing forward/backward passes natively on the hardware.

The project's README lays it bare: this is a research proof-of-concept, not a polished library. It includes Objective-C code to set up ANE sessions, load models, and run training loops on small networks like MLPs or tiny CNNs. Benchmarks reveal intriguing hardware traits—throughput hitting peaks under ideal conditions, SRAM utilization patterns, and power draw that's impressively efficient compared to CPU fallbacks. For instance, on M-series chips, it clocks matrix ops at speeds rivaling inference tools, hinting at untapped potential.

What makes this technically fascinating? The ANE's architecture—fixed-function units for common NN ops—has always screamed "training capable," but software barriers prevented it. By reverse-engineering, maderix proves the silicon can handle gradients and updates, exposing challenges like low peak utilization (currently ~2-3%) and ops falling back to CPU for element-wise math. It's a wake-up call: NPUs like ANE aren't inference toys; they're general-purpose ML accelerators waiting for bolder software.

This isn't replacing PyTorch or JAX for large models—yet. Training stays experimental, suited to toy datasets or research spikes. But it changes the conversation for on-device ML. Imagine fine-tuning models locally without cloud dependency, or optimizing edge AI on battery-constrained devices. As a reference, it arms low-level hackers to probe ANE limits, potentially inspiring open frameworks or pressuring Apple for official support.

Gaining explosive traction in days, the repo has sparked benchmarks across hardware generations, from A17 Pro to M3 Ultra. Forks are dissecting SRAM quirks and op coverage, while discussions debate scalability. For Apple ML builders, it's a manifesto: hardware abundance meets software scarcity. The real story? ANE training works—now the engineering marathon begins.

It surfaces entity relationships without interpretation, scoring, or bias, empowering users to uncover patterns themselves.

At its core, br-acc runs 45 ETL pipeline modules, tracked in docs/source_registry_br_v1.csv with statuses like loaded, partial, stale, blocked, or not built. These pipelines ingest raw data from outlets like CNPJ (business registry), TSE (electoral court), DOU (official gazette), PGFN (tax debts), and Transparência (spending). Normalized entities—companies, officials, contracts—form a graph schema with loaders and query surfaces. Builders get reproducibility via a one-command local bootstrap: cp .env.example .env && make bootstrap-demo, spinning up Docker services for Neo4j, a FastAPI public API, and React frontend. Check health at http://localhost:8000/health, explore at http://localhost:3000, or browse the graph at http://localhost:7474.

The project, just two days old with over 1,000 stars, emphasizes privacy: LGPD-compliant, no personal data, public-safe defaults. A full ingestion orchestration handles heavy lifts via make targets, supporting bring-your-own-data workflows.

Latest release v0.4.0 hardens the stack for production and collaboration:

This matters to builders tackling civic tech, open data graphs, or federated ETL. It models decentralized data unification without vendor lock-in, using familiar tools like Neo4j, FastAPI, React, and Docker. Contributions target pipeline gaps or query extensions, tracked in CSV. PythonRobotics Delivers Readable Python Codes for Core Robotics Algorithms

A decade-old repository provides textbook-style implementations of localization, SLAM, path planning, and control for robot developers.

AtsushiSakai/PythonRobotics Python 28.8k stars

In the demanding field of robotics, where algorithms for navigation and control must balance theory with real-world deployment, PythonRobotics stands out as a focused collection of Python implementations. Created by Atsushi Sakai in 2016, this repository functions as both code library and textbook, emphasizing clarity over complexity. It targets essential problems: localizing a robot in unknown environments, building maps from sensor data, planning collision-free paths, and tracking those paths with precise control.

The project organizes algorithms into intuitive categories. Localization includes Extended Kalman Filter (EKF), Particle Filter, and Histogram Filter, each with self-contained Python scripts that visualize the process. For mapping, developers find Gaussian grid maps, ray casting, LiDAR-to-grid conversion, and even k-means clustering for object detection or rectangle fitting. SLAM coverage features Iterative Closest Point (ICP) matching and FastSLAM 1.0, enabling simultaneous localization and mapping without heavy frameworks.

Path planning forms the repository's backbone, tackling static and dynamic environments. Grid-based methods like Dijkstra, A*, D*, and D* Lite handle known spaces efficiently. Sampling-based planners such as Rapidly-Exploring Random Trees (RRT), RRT*, and Probabilistic Road-Map (PRM) excel in high-dimensional configurations, with variants like LQR-RRT* and Reeds-Shepp paths for non-holonomic vehicles. Dynamic approaches include Dynamic Window Approach (DWA), Potential Fields, and State Lattice Planning with lane or biased polar sampling. Optimization techniques like Particle Swarm Optimization (PSO) and model predictive control round out options for autonomous driving and navigation.

Path tracking scripts implement Stanley control, LQR-based speed and steering, and advanced Nonlinear Model Predictive Control (NMPC) using C-GMRES. Specialized sections cover arm navigation with obstacle avoidance, drone 3D trajectory following, rocket-powered landing, and even bipedal planning via inverted pendulum models.

What sets PythonRobotics apart is its design philosophy: algorithms are selected for practicality and readability, with minimum dependencies—often just NumPy and Matplotlib for animations. Comprehensive documentation, including a "Getting Started" guide and YouTube overview, makes it ideal for self-study. With nearly 29,000 stars reflecting steady community interest over a decade, it serves builders prototyping autonomous vehicles, drones, or manipulators, bridging academic concepts to deployable code.

Builder's Take Start with the documentation and YouTube overview to grasp the structure before running scripts. Expect pure Python with light dependencies, so it's perfect for Jupyter experimentation but may need integration tweaks for production ROS or real-time systems. Fork and contribute via established guidelines if extending for specific hardware.
