Codex Skill Automates Full Paper-Cut Ad Production from Script to MP4 🔗

Integrates voice cloning, animation, and quality checks into a single OpenAI Codex workflow for localized ad creation

Jane-xiaoer/paper-collage-ad-codex · JavaScript · 265 stars 5d old

Jane-xiaoer/paper-collage-ad-codex is a newly released OpenAI Codex skill that automates the end-to-end production of paper-cut collage advertisements — from initial concept and scriptwriting through animation, voiceover, sound design, and final MP4 quality validation. Built for developers and creative teams working within the Codex ecosystem, it transforms a simple natural language prompt into a polished, broadcast-ready 45-second ad, all executed locally on Apple Silicon Macs.

At its core, the skill functions as a creative pipeline orchestrator.

When a user invokes it with a command like “用 paper-collage-ad 给这个产品制作一条有趣的 45 秒剪纸广告。”, Codex reads the skill’s SKILL.md and activates a sequence of modular components: it extracts a visual metaphor from product assets, generates a timed script and storyboard, produces style-locked keyframes using real brand materials, animates them via Seedance, HyperFrames, layered PNGs, or FFmpeg, and layers in narration, music, foley, and SFX. The final output is an H.264/AAC MP4 that passes stream-level validation — ready for deployment without further editing.

What sets this apart is its integration of local voice cloning through IndexTTS-2 MLX, a lightweight, zero-sample voice synthesis tool optimized for Apple Silicon. Rather than relying on cloud-based TTS APIs with privacy and latency concerns, the skill enables users to clone licensed voices using just 6–12 seconds of reference audio. The process involves generating a speaker embedding (speaker-v2.npz) locally, then synthesizing 48 kHz WAV narration for each scene — all stored privately within the user’s project. This approach avoids uploading voice data and supports emotional tone control, making it suitable for branded content where vocal consistency and data sovereignty matter.

The skill is designed to be dependency-transparent: core functions like static keyframe generation, layered animation, and final compositing require only ffmpeg, node, and bash — no API keys needed. Optional integrations with Seedance,即梦 (Jimo), MiniMax, and ElevenLabs remain available for users who prefer those services, but the default path stays fully offline and self-contained. Installation is straightforward via global or project-scoped cloning into ~/.codex/skills/, with clear workflow documentation in Chinese (though the code and scripts are language-agnostic).

Despite its rapid adoption — 267 stars and 35 forks in just five days — the project raises a practical concern for broader adoption: its current scope is tightly coupled to macOS and Apple Silicon due to reliance on MLX-accelerated IndexTTS-2. While the README notes that core animation and compositing work cross-platform, the voice cloning feature — a highlighted capability — is explicitly positioned as “suitable for Apple Silicon Mac.” This creates a platform split: teams on Linux or Windows would need to forego the local voice cloning advantage or implement workarounds, potentially fragmenting the user experience. Whether the skill will expand its voice pipeline to support cross-platform runtimes remains an open question as it evolves beyond its initial release.

The catch: The voice cloning feature, a key differentiator, currently requires Apple Silicon and MLX, limiting accessibility for developers on Linux or Windows who want the full end-to-end local workflow.

  • Indie developers automating ad production for app store listings
  • Educators generating paper-cut style explainers with custom narration

Aether Offers Stealthy Network Tunneling for Restricted Environments 🔗

A Rust-based client that evades censorship via HTTP/3, WireGuard, and traffic obfuscation with automatic route discovery

CluvexStudio/Aether · Rust · 244 stars 0d old

Aether is a user-space proxy client designed to bypass network restrictions in environments where traditional VPNs fail. Built in Rust, it counters Deep Packet Inspection (DPI), protocol fingerprinting, UDP throttling, and endpoint blocking by automatically discovering reachable routes and establishing encrypted tunnels. Unlike conventional tools, Aether supports MASQUE over HTTP/3 and HTTP/2, WireGuard (including a nested mode called gool), and traffic obfuscation techniques to blend in with normal web traffic.

Upon launch, the client prompts users to select a protocol, obfuscation profile, listening port, and scan mode. It then exposes a local SOCKS5 proxy at 127.0.0.1:1819, allowing browsers, terminals, or system-wide proxy settings to route traffic through the tunnel. The latest release, v1.0.1, changed the default SOCKS5 port from 10808 to 1819 (overridable via AETHER_SOCKS environment variable) and introduced aether.sh, a one-line Termux installer that auto-detects device architecture, downloads the correct binary, verifies its SHA-256 checksum, and installs it into $PREFIX/bin.

Prebuilt binaries are available for Linux (x86_64, arm64, armv7), macOS (Intel and Apple Silicon), Windows (x86_64 and arm64), and Android via Termux. Each release includes individual .sha256 files and a combined SHA256SUMS.txt for integrity verification. The project requires Rust, a C/C++ compiler, and CMake to build, with the quiche repository expected alongside the source for MASQUE support.

Aether’s strength lies in its automation and adaptability: it doesn’t rely on pre-configured endpoints but dynamically probes for working paths, making it resilient in volatile, heavily filtered networks. For developers and sysadmins operating in or supporting users within restrictive regimes, it offers a programmable, open-source alternative to opaque commercial circumvention tools.

The catch: As a 1-day-old project with 8 open issues and a narrow focus on user-space tunneling, Aether lacks long-term stability data, enterprise audits, and clarity on how its obfuscation profiles hold up against advanced, adaptive DPI systems at scale.

Use Cases
  • Developers accessing blocked APIs in censored regions
  • Sysadmins maintaining remote tools behind national firewalls
  • Researchers testing network resilience under protocol-level restrictions

PyTorch 2.13.0 Boosts Distributed Training with Fault-Tolerant Comms 🔗

New torchcomms backend and FlexAttention gains target large-scale model efficiency on modern hardware

pytorch/pytorch · Python · 101.8k stars Est. 2016 · Latest: v2.13.0

PyTorch’s latest release, v2.13.0, sharpens its focus on scalable AI infrastructure with two key advances: a redesigned communications layer for distributed training and expanded hardware support for its FlexAttention mechanism.

The new torchcomms backend replaces legacy MPI-based collectives in PyTorch Distributed, aiming to improve fault tolerance, scalability, and debuggability when training models across thousands of GPUs. By decoupling communication logic from the training loop and adding built-in retry mechanisms, torchcomms reduces silent failures and simplifies diagnostics in large-cluster environments — a persistent pain point for teams running LLMs or multimodal models at scale.

On the performance front, FlexAttention — PyTorch’s programmable attention kernel — now runs on Apple Silicon via the MPS backend, delivering up to 12x speedups over standard scaled dot-product attention (SDPA) for sparse patterns. Crucially, it also gains a deterministic backward pass on CUDA, enabling reproducible gradient computation essential for research and debugging. Complementing this, the CuTeDSL "Native DSL" backend prototype gives TorchInductor an alternative compilation path to Triton, promising faster kernel generation for GPU ops without sacrificing performance.

Memory efficiency gets a boost too: the new nn.LinearCrossEntropyLoss module fuses the final linear layer and loss calculation, cutting peak GPU memory usage by up to 4x during large-vocabulary language model training — a significant saving when scaling to trillion-parameter architectures. Python 3.15 wheel support further modernizes the deployment pipeline, aligning PyTorch with the latest language runtime.

Despite these strides, the project’s rapid evolution surfaces tension in its release cadence. With over 18,000 open issues and a community-driven development model, builders must weigh innovation against stability. The catch: while torchcomms and FlexAttention push PyTorch toward exascale readiness, their prototype status and shifting APIs mean production teams may need to pin versions or fork internal forks to avoid breakage during active development.
